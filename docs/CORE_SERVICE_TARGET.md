# 核心服务目标

> 这个语言存在的理由

---

## 一句话定位

**自然语言意图 → 可证明、可执行、可进化的电路级代码**

---

## 核心服务目标

### 🎯 目标用户：AI Agent 开发者

需要一个能"把自然语言意图直接变成生产级电路代码"的编程语言和工具链。

### 🔥 核心痛点

1. **LLM 生成的代码无法保证正确性**——幻觉约束、未验证假设、未定义行为
2. **自然语言意图与生成的代码之间语义断裂**——设计师说"安全"，代码不一定真的安全
3. **验证靠测试，不靠证明**——测试覆盖率再高，也没法证明"这件事永远不会错"
4. **多 Agent 协作时意图无法跨 Agent 追踪**——同一个意图在不同 Agent 之间出现不同的理解

### ✨ 我们的解决方案

- **CNL 约束规约**：用 Racket DSL 将自然语言意图编码为数学上可验证的约束表达式
- **Rosette 形式证明**：在编译期验证约束是否被正确实现，而不是靠运行时测试
- **双峰塔架构**：Racket 前端保持思想的纯粹性，C++ 后端保证执行的可靠性
- **Datalog 知识图谱**：意图历史全局可追踪，跨 Agent 协作零误解

---

## 典型使用场景

### 场景 1：人类设计师描述电路规格 → 形式化约束 → 验证实现

```
设计师："我要一个加法器，输入非空、位宽匹配，输出维度一致"
    ↓ CNL 约束规约
Rosette 验证："加法器实现是否满足输入非空、位宽匹配、输出维度一致？"
    ↓ 形式化证明
C++ 后端生成机器码：通过验证，零运行时开销
```

### 场景 2：多 Agent 协作 → 意图共享 → 零重复犯错

```
Agent A 定义意图 → 存入 Datalog 图谱
Agent B 检索相似意图 → 避免重复设计
Agent C 执行验证 → 结果同步回图谱
```

### 场景 3：AI 生成代码的自动校验闭环

```
LLM 生成电路 DSL 代码
    ↓
前端注入 Contracts（C++26 风格）
    ↓
Rosette 形式验证
    ↓
通过 → 编译成 C++ 后端机器码
失败 → 返回错误位置和违反的约束
```

## 服务边界

### ✅ 我们解决的问题

- 意图的形式化规约与验证
- 自然语言 → 电路 DSL 的完整转换
- 多 Agent 意图追踪与协作
- 生产级代码的可证明正确性

### ❌ 我们不解决的问题

- 通用 AI 模型的训练和推理（那是模型厂商的事）
- 非电路领域的通用编程（我们有专注点）
- 硬件设计本身的 EDA 工具链（我们输出代码，不是替代 EDA）

---

## 成功标准

当一个设计师说"我要一个安全的高性能加法器"，我们能：

1. **完整捕获**其意图（输入约束、性能约束、安全约束）
2. **形式化证明**生成的代码满足这些约束
3. **生成可部署**的 C++ 代码，零运行时额外开销
4. **全局追踪**这个意图的验证历史和实现版本

---

## 与现有方案的对比

| 维度 | 传统 VHDL/Verilog | LLM 直接生成 | **我们的方案** |
|------|------------------|--------------|---------------|
| 意图表达 | 手动写代码 | 自然语言（无验证） | CNL 约束规约 |
| 正确性保证 | 仿真测试 | 无保证 | Rosette 形式证明 |
| 可追踪性 | 版本管理 | 无 | Datalog 知识图谱 |
| 生产部署 | 直接可用 | 需人工审查 | C++ 后端直接部署 |
| 多 Agent 协作 | 不支持 | 不支持 | 原生支持 |
