# 03 — 编译器可验证：Contracts

## 问题

语言的核心承诺是"代码可验证"。但这个承诺成立的前提是：**编译器本身是正确的**。

- 宏展开器不能引入语义错误
- 增量编译的依赖图不能有环或漏边
- 代码生成器不能产生未定义行为

如果编译器有 bug，所有经过它编译的"已验证代码"都不可信。

## Contracts 解决什么

C++26 Contracts 允许在代码中声明可编译期检查的前置/后置/不变量条件：

```cpp
[[assert: expr]]               // 断言表达式在编译期/运行时检查
[[expects: condition]]         // 函数前置条件
[[ensures: condition]]         // 函数后置条件
```

用于编译器内部：
- 宏展开器的输入/输出类型约束
- 增量编译依赖图的拓扑排序合法性
- 代码生成阶段的 IR 不变量检查

## 与语言核心的对应

```
语言保证"AI 写的代码可验证"
    ↓ 编译器自身也需要相同级别的保证
Contracts → 编译器的内部不变量 → 编译器自身可验证
```

## 核心思路

不是"在生成的代码里加 Contracts"——是 **编译器的每一层内部都用 Contracts 自检**。宏展开器输出 AST 的合法性、增量编译缓存的正确性、代码生成不违反优化假设——这些不变量属于编译器，应该由编译器自己保证。

信任链条：Contracts（编译器正确）→ 增量编译（变换正确）→ 语言代码（语义正确）。
