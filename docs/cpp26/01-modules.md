# 01 — 增量编译基座：Modules

## 问题

从最小内核开始生长的语言，编译器本身会不断膨胀：reader、宏展开器、类型检查器、增量编译追踪器、代码生成器……每次扩展语言能力，编译器也需要变化。

如果编译器是单体的，改动一个模块就触发全量重编译——这对"自动修复、变换、增量编译"的进化闭环是致命障碍。

## Modules 解决什么

C++26 Modules 提供：

- **显式的模块接口**（`.cppm`）和实现分离——编译器各个组件之间有明确的依赖边界
- **编译期的模块依赖图**——只重编译变化的模块及其下游，不是整个编译器
- **宏/符号隔离**——模块内定义的宏不会泄露到外部，减少意外耦合

## 与语言核心的对应

```
语言的增量编译（AI 修复/变换 AST → 增量验证）
    ↓ 编译器本身也需要增量编译
C++26 Modules → 编译器的模块化依赖图 → 编译器的增量重编译
```

语言的进化机制（增量编译）和编译器的进化机制（Modules）是同类设计——都是通过最小化变动范围来实现快速迭代。

## 核心思路

不是"用 C++ Modules 写程序"——是 **用 C++ Modules 构建一个能跟着语言一起生长的模块化编译器**。

编译器不是一次性成品，是和语言共同进化的活系统。
