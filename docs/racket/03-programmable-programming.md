# 03 — 可编程编程：制造语言的语言

前两篇讲的是"能做什么"。这篇讲的是"为什么这是一种世界观"。

## 传统方法

解决一个问题：设计接口、写库、调函数。

层数越多，抽象越厚，底层被埋得越深。C 写了 OS，C++ 在其上，Python 在其上，框架在其上——每一层都是对下一层的隐藏。出问题时，需要穿透所有层才能理解。

## 可编程编程的方法

不对问题进行抽象封装。**为问题设计一种语言，用语言直接表达问题。**

```
问题：表达网络包处理
传统：写一个 parse_packet() 函数，返回 struct
可编程编程：设计一种语言，其中 packet 是第一等公民

#lang packet-lang
(defrule http-request
  (match method path version)
  (parse headers)
  (process body))
```

区别不在于语法，而在于**问题的结构被语言直接捕获，而不是被函数调用间接表达**。

## 对 AI 意味着什么

AI 不需要在脑子里做"意图 → 函数调用"的翻译。

AI 可以直接设计语言的语法，让意图的结构成为代码的结构。

```
人类意图：一个确认写操作在日志落盘后才返回的存储系统
语言设计：引入 ensure-durable 原语作为语言构造
          编译器保证这个构造的所有路径都经过 fsync
```

函数调用可以被绕开、可以被忘记。语言构造不行——它是语法的一部分，编译器/宏展开器对所有路径可见。

## 与传统抽象的分界线

| | 传统抽象 | 可编程编程 |
|--|---------|-----------|
| 方式 | 封装、隐藏 | 暴露、命名 |
| 工具 | 函数、类、接口 | 语言构造、宏、#lang |
| 验证 | 测试覆盖 | 语法保证（因为不可能写出违反规则的代码） |
| 消费 | 开发者调用 API | AI 直接操作语义原语 |

## 总结

可编程编程不是"用 Racket 写 DSL"。它是一句断言：

**任何值得解决的复杂问题，都值得为它设计一种语言。**

AI 的语言工厂。不是用工具解决问题——是制造解决问题的工具。
